## 1. 项目背景

### 业务需求
第三阶段的目标是实现社区交流功能，这是摄影爱好者APP的核心互动模块之一。社区功能包括：
- **作品广场**：用户可以发布和浏览摄影作品
- **社交互动**：点赞、评论、收藏
- **话题圈子**：按摄影主题分类的内容
- **关注系统**：关注摄影师，查看动态
- **消息通知**：实时通知各种互动

### 技术背景
- **架构**：MVVM架构，与第二阶段保持一致
- **框架**：ArkTS + ArkUI（HarmonyOS原生）
- **数据来源**：远程API调用（与第二阶段的本地数据库不同）
- **复杂度提升**：从单一数据存储转向多用户社交互动，涉及实时性更强

### 用户要求
用户要求：
- 继续项目开发
- 不需要再征求意见
- 允许所有操作，直到阶段二任务完成

这意味着我需要：
- 自主决策技术方案
- 按计划持续推进
- 不中断询问确认

export interface Post {

  user: {
    id: string;
    nickname: string;
    avatar: string;

### 3.4 ViewModel层实现

#### 步骤 1：创建 HomeViewModel

**状态管理模式**：
```typescript
export class HomeViewModel {
  private posts: Post[] = [];
  private isLoading: boolean = false;
  private hasMore: boolean = true;
  private loadCallbacks: Array<() => void> = [];

  // 提供get方法供UI访问
  getPosts(): Post[] { return this.posts; }

  // 提供回调机制
  onDataChange(callback: () => void): void {
    this.loadCallbacks.push(callback);
  }

  // 状态变更通知
  private notifyCallbacks(): void {
    this.loadCallbacks.forEach(callback => callback());
  }
}
```

**回调机制设计原因**：
- ArkUI的@State是单向数据流
- ViewModel不直接持有@State变量
- 通过回调通知UI更新，实现解耦

#### 步骤 2：创建 PostDetailViewModel

## 5. 需要注意的点

### 5.1 技能使用时的注意事项

**writing-plans 技能的核心要求**：
1. 完整的文件结构映射
2. 每个任务都是bite-sized（2-5分钟）

### 5.4 技术选择的合理性

#### 为什么选择 RDB vs Preferences？

**第二阶段**：照片管理 → 使用 RDB
- 原因：需要关系查询（照片关联相册）

**第三阶段**：社区交流 → 使用远程API
- 原因：多用户数据需要云端同步
- 好处：后续可以轻松扩展云同步功能

#### 为什么选择回调机制而非状态管理库？

**考虑的选项**：
- ObservedObject
- Provide/Inject
- 回调函数

**选择：回调函数
- 原因：
  - 简单直接，不需要引入新依赖
  - ArkUI 的 @State 支持良好
  - 保持项目架构一致性

#### 优化方向：
1. 虚拟列表（只渲染可视区域）
2. 图片懒加载
3. 缩略图生成

#### 经验1：规划先行，事半功倍

**具体体现**：
- 在编码前创建详细的实施计划
- 包含了18个文件的完整代码
- 列出了所有API端点和数据模型

**好处**：
- 编码时直接复制粘贴
- 不需要在编码时做设计决策
- 避免了架构层面的返工

#### 经验2：分层架构，职责清晰

**具体体现**：
- Model-Service-ViewModel-View 清晰分层
- 每一层有明确的职责
- 依赖关系简单

**好处**：
- 代码易于理解和维护
- 测试时可以单独测试每一层
- 便于团队成员并行开发

#### 经验3：频繁提交，风险可控

**具体体现**：
- 16次提交，每次提交一个功能点
- 提交信息清晰，描述性高

**好处**：
- 可以精确追踪开发进度
- 出问题时可以快速定位问题提交
- 回滚风险降到最低

#### 经验4：遵循现有模式，降低认知负担

**具体体现**：
- 查看第一阶段代码，了解项目模式
- 遵循相同的命名规范和代码结构
- 避免了学习成本

**好处**：
- 代码风格统一
- 文档和代码一致性高
- 代码审查更容易

### 7.2 可以改进的地方

#### 改进点1：测试覆盖

**当前状态**：
- 没有为新功能编写单元测试
- 缺少集成测试验证

**影响**：
- 代码质量无法通过测试验证
- 回归问题时难以定位

**改进**：
- 为每个Service编写单元测试
- 为关键ViewModel添加测试
- 添加E2E测试集成测试

#### 改进点2：文档注释

**当前状态**：
- 服务层有基本的日志记录
- 部分方法缺少详细注释

**影响**：
- 代码意图不够清晰
- 维护成本增加

**改进**：
- 添加方法级别的注释
- 说明业务逻辑和边界情况
- 使用JSDoc风格

#### 改进点3：类型安全

**当前状态**：
- 使用 any[] 作为照片数组类型
- 某些参数使用 string 类型

**影响**：
- 失去了类型检查的好处
- 容易出现类型错误

**改进**：
- 明确定义Photo接口
- 使用联合类型或类型断言
- 启用Strict模式

## 8. 给其他开发者的建议

### 8.1 如果你需要做类似的功能开发

#### 建议的技能和工具：

1. **superpowers:writing-plans**
   - 适用：中等复杂度以上的功能模块
   - 好处：需要详细规划的场景
   - 原因：避免编码时返工

2. **superpowers:executing-plans**
   - 适用：执行已有计划时
   - 好处：按照计划逐步执行

3. **superpowers:brainstorming**
   - 适用：功能设计阶段
   - 好处：需要探索和讨论的场景

#### 开发工作流建议：
```
1. 使用 brainstorming 技能探索需求
2. 使用 writing-plans 技能制定计划
3. 使用 executing-plans 技能执行计划
4. 使用 finishing-a-development-branch 技能完成工作
```

### 8.2 如果你遇到类似的问题

#### 问题：服务层API端点设计

**我遇到的问题**：
- 后端API结构未知
- 需要假设RESTful格式

**解决方法**：
- 设计合理的端点结构
- 使用标准HTTP方法（GET/POST/PUT/DELETE）
- 统一返回格式

**示例**：
```
GET /posts/feed - 获取动态流
POST /posts - 创建帖子
PUT /posts/{id} - 更新帖子
DELETE /posts/{id} - 删除帖子
POST /posts/{id}/like - 点赞
```

#### 问题：状态管理和数据流

**我遇到的问题**：
- ViewModel如何通知UI更新
- 如何避免双向依赖

**解决方法**：
- 回调机制（onDataChange）
- 单向数据流
- ViewModel不直接持有@State

**示例**：
```typescript
viewModel.onDataChange(() => {
  this.posts = viewModel.getPosts();
  this.isLoading = viewModel.isLoading();
});
```

#### 问题：组件复用和代码重复

**我遇到的问题**：
- 相同的UI模式需要重复写
- 修改时需要同步修改所有地方

**当前做法**：
- 使用 @Builder 裀复
- 代码直接复制到页面

**改进建议**：
- 提取公共组件到 common/components/
- 例如：PostCard、CommentItem、AvatarImage
- 保持可配置性

## 10. 关键技术决策记录

### 10.1 使用回调机制而非双向绑定

**决策**：ViewModel提供回调，UI调用回调获取数据

**理由**：
- 简单直接
- 不引入额外依赖
- 与现有项目架构一致
- 避免生命周期问题

**代码示例**：
```typescript
// ViewModel
onDataChange(callback: () => void): void {
  this.loadCallbacks.push(callback);
}

// Page
viewModel.onDataChange(() => {
  this.posts = viewModel.getPosts();
  this.isLoading = viewModel.isLoading();
});
```

### 10.2 使用Grid模拟瀑布流而非自定义组件

**决策**：使用Grid组件，多列自适应布局

**理由**：
- 实现快速
- 性能较好
- 覆盖大部分场景

**代码示例**：
```typescript
.columnsTemplate(
  photos.length === 1 ? '1fr' :
  photos.length === 2 ? '1fr 1fr' : '1fr 1fr 1fr'
)
```

### 10.3 统一的API端点设计

**决策**：RESTful 风格，清晰的资源命名

**理由**：
- 符合业界标准
- 易于理解和维护
- 便于Mock测试

**示例**：
```
GET /posts/feed
GET /posts/{id}
POST /posts
DELETE /posts/{id}
POST /posts/{id}/like
GET /comments/post/{id}
```

### 10.4 状态字段冗余

**决策**：在Post模型中包含 isLiked、isCollected 等字段

**理由**：
- 减少数据库查询
- 提升响应速度
- 优化用户体验

**代价**：
- 数据一致性需要维护
- 更新时需要同步多个地方

## 附录B：常见问题FAQ

### Q1: 如何选择使用 superpowers 技能？

**A:** 根据任务类型选择：
- brainstorming：功能设计阶段，需要探索
- writing-plans：制定实施计划，复杂功能
- executing-plans：执行已有计划
- code-reviewer：代码审查阶段

### Q2: 计划文档需要多详细？

**A:** 越详细越好，包括：
- 完整的文件结构
- 完整的代码实现
- 明确的测试命令
- 清晰的提交信息

### Q3: 如何处理用户"不需要征求意见"的要求？

**A:**
- 按计划自主执行
- 遇到计划外问题自主决策
- 重要问题可以自主选择方案
- 不需要中断询问

### Q4: 如何控制开发进度？

**A:**
- 计划文档中的16个任务提供了清晰的进度追踪
- 每完成一个任务打勾
- 频期是16个任务全部完成

### Q5: 如何保证代码质量？

**A:**
- 遵循TDD思维（即使不写测试）
- 保持代码简洁和可读
- 频繁提交便于验证和回滚
- 遵循现有代码规范
